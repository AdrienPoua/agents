# Architecture du dossier .claude

Ce document explique l'organisation du dossier `.claude` et le rôle de chaque sous-dossier dans la configuration de Claude Code.

## Vue d'ensemble

Le dossier `.claude` contient la configuration personnalisée pour Claude Code. Il permet de structurer et d'organiser différents types d'outils et de workflows spécialisés.

```
.claude/
├── agents/              # Agents spécialisés
├── commands/            # Slash commands personnalisés
├── prompts/             # Templates de prompts
├── hooks/               # Documentation des hooks
├── docs/                # Documentation interne
└── settings.local.json  # Configuration locale
```

---

## Structure détaillée

### agents/

**Rôle** : Contient des agents spécialisés pour des tâches complexes et multi-étapes.

**Quand utilisé** : **À LA DEMANDE** - Uniquement lorsque vous invoquez explicitement un agent.

**Format** : Fichiers Markdown (`.md`) contenant les instructions détaillées pour l'agent.

**Exemple d'utilisation** :

```
User: "Utilise l'agent Clean Architecture pour implémenter la feature X"
→ Claude charge .claude/agents/clean-architecture.md
→ Claude suit les instructions de l'agent pour la tâche
```

**Cas d'usage** :

- Workflows complexes nécessitant plusieurs étapes (Clean Architecture, Supabase, Sanity)
- Tâches répétitives avec une méthodologie spécifique
- Domaines techniques nécessitant une expertise particulière

**Structure recommandée d'un agent** :

```markdown
# Agent Name

## Role

[Description du rôle de l'agent]

## When to use

[Quand utiliser cet agent]

## Process

1. Step 1
2. Step 2
3. Step 3

## Examples

[Exemples concrets]
```

---

### commands/

**Rôle** : Contient des slash commands personnalisés qui se transforment en prompts détaillés.

**Quand utilisé** : **À LA DEMANDE** - Lorsque vous tapez une slash command (ex: `/epct`, `/feature`).

**Format** : Fichiers Markdown (`.md`) qui sont "expandés" en prompts au moment de l'invocation.

**Exemple d'utilisation** :

```
User: "/feature ajouter un système de notifications"
→ Claude lit .claude/commands/feature.md
→ Le contenu devient le prompt complet
→ Claude exécute le workflow défini dans feature.md
```

**Cas d'usage** :

- Workflows récurrents (EPCT: Explore, Plan, Code, Test)
- Processus métier standardisés (ajouter une feature, refactoring)
- Séquences d'actions prédéfinies

**Différence avec les agents** :

- **Commands** : Transformations de prompt, workflows courts et directs
- **Agents** : Comportements complexes avec logique conditionnelle et multi-étapes

---

### prompts/

**Rôle** : Templates réutilisables pour générer du code ou des structures spécifiques.

**Quand utilisé** : **À LA DEMANDE** - Lorsque vous référencez explicitement un prompt dans votre message.

**Format** : Fichiers Markdown (`.md`) contenant des templates et des patterns de code.

**Exemple d'utilisation** :

```
User: "Crée un composant en suivant prompts/component.md"
→ Claude lit .claude/prompts/component.md
→ Claude génère le code selon le template défini
```

**Cas d'usage** :

- Patterns de composants React
- Templates de Server Actions Next.js
- Structures de fichiers standardisées
- Conventions de code spécifiques au projet

**Différence avec les commands** :

- **Prompts** : Templates passifs, guidelines de structure
- **Commands** : Workflows actifs, séquences d'actions

---

### hooks/

**Rôle** : Documentation sur les hooks d'automatisation (Git hooks, file hooks).

**Quand utilisé** : **DOCUMENTATION** - Ces fichiers ne sont **pas lus automatiquement** par Claude Code.

**Format** : Fichiers Markdown (`.md`) expliquant la configuration et l'utilisation des hooks.

**Contenu typique** :

- Explication des hooks Git (pre-commit, commit-msg)
- Configuration des hooks de fichiers dans `settings.local.json`
- Exemples de workflows automatisés
- Troubleshooting des hooks

**Note importante** : Les hooks réels sont configurés dans `settings.local.json` (section `hooks`), ces fichiers servent uniquement de documentation pour l'utilisateur.

---

### docs/

**Rôle** : Documentation interne du dossier `.claude` et guides d'utilisation.

**Quand utilisé** : **DOCUMENTATION** - Fichiers de référence pour l'utilisateur, non lus par Claude Code.

**Format** : Fichiers Markdown (`.md`) de documentation.

**Contenu typique** :

- README.md : Vue d'ensemble de la structure .claude
- quick-start.md : Guide de démarrage rapide
- .claude-doc.md (ce fichier) : Architecture détaillée du dossier .claude

**Audience** : Développeurs humains, nouveaux membres de l'équipe, documentation de référence.

---

### settings.local.json

**Rôle** : Fichier de configuration principal de Claude Code pour ce projet.

**Quand utilisé** : **AU DÉMARRAGE** - Lu automatiquement par Claude Code au début de chaque conversation.

**Format** : JSON conforme au schéma `claude-code-settings.json`.

**Contenu typique** :

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "model": "sonnet",
  "permissions": {
    "allow": ["SlashCommand(/epct:*)", "Bash(pnpm dev:*)"],
    "deny": ["Bash(rm -rf:*)"],
    "ask": ["Bash(git push:*)"]
  },
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [{ "type": "command", "command": "pnpm fix" }]
      }
    ]
  }
}
```

**Sections principales** :

- **model** : Modèle Claude à utiliser (sonnet, opus, haiku)
- **permissions** : Contrôle des commandes autorisées/refusées/nécessitant confirmation
- **hooks** : Automatisations déclenchées après certaines actions (Write, Edit, etc.)

## Cycle de vie des éléments .claude

### Lecture automatique (au démarrage)

- ✅ `settings.local.json` - Configuration chargée au début de chaque conversation

### Lecture à la demande (invocation explicite)

- ✅ `agents/*.md` - Chargés quand vous demandez explicitement un agent
- ✅ `commands/*.md` - Expandés quand vous tapez une slash command
- ✅ `prompts/*.md` - Lus quand vous référencez un prompt dans votre message

### Documentation uniquement (lecture manuelle)

- ❌ `hooks/*.md` - Documentation pour configurer les hooks
- ❌ `docs/*.md` - Guides et documentation de référence

## Bonnes pratiques d'organisation

### 1. Nommage des fichiers

**Agents** : Nom du domaine ou de la technologie

```
agents/
├── clean-architecture.md
├── supabase.md
└── sanity.md
```

**Commands** : Verbe d'action ou nom du workflow

```
commands/
├── epct.md         # Explore, Plan, Code, Test
├── feature.md      # Ajouter une nouvelle feature
└── refactor.md     # Refactoring guidé
```

**Prompts** : Type de code ou structure

```
prompts/
├── component.md        # Composants React
├── server-action.md    # Server Actions Next.js
└── use-case.md         # Use Cases Clean Architecture
```

### 2. Structure d'un fichier d'agent

```markdown
# [Agent Name]

## Rôle

[1-2 phrases décrivant le rôle]

## Quand l'utiliser

[Contextes d'utilisation]

## Prérequis

[Ce qui doit être en place avant d'utiliser l'agent]

## Processus

1. Étape 1
2. Étape 2
3. Étape 3

## Exemples

[Exemples concrets avec code]

## Troubleshooting

[Problèmes courants et solutions]
```

### 3. Structure d'une slash command

```markdown
# Command: /[command-name] [args]

## Description

[Ce que fait la commande]

## Syntaxe

`/[command-name] [arg1] [arg2?]`

## Workflow

1. Première action
2. Deuxième action
3. Troisième action

## Exemples

- `/command exemple1`
- `/command exemple2 --option`
```

### 4. Structure d'un prompt

```markdown
# [Template Name]

## Objectif

[Ce que ce prompt génère]

## Principes

1. Principe 1
2. Principe 2

## Structure

[Structure de fichier attendue]

## Exemple complet

\`\`\`typescript
// Code exemple
\`\`\`

## Anti-patterns

❌ [Ce qu'il ne faut PAS faire]
```

## Workflows recommandés

### Workflow 1 : Ajouter une nouvelle feature

```
1. User: "/feature système de notifications"
   → Charge .claude/commands/feature.md
   → Workflow d'exploration et planification

2. User: "Utilise l'agent Clean Architecture"
   → Charge .claude/agents/clean-architecture.md
   → Implémentation guidée par l'agent

3. User: "Crée les composants selon prompts/component.md"
   → Charge .claude/prompts/component.md
   → Génération du code selon le template
```

### Workflow 2 : Refactoring guidé

```
1. User: "/refactor améliorer la performance"
   → Charge .claude/commands/refactor.md
   → Analyse et propositions

2. User: "Suis les patterns dans prompts/component.md"
   → Charge .claude/prompts/component.md
   → Refactoring selon les standards
```

## Maintenance

### Quand ajouter un nouvel élément ?

**Nouvel agent** :

- Tâche complexe répétée 3+ fois
- Workflow multi-étapes avec décisions conditionnelles
- Domaine technique nécessitant expertise spécifique

**Nouvelle command** :

- Workflow récurrent standardisé
- Séquence d'actions prévisible
- Besoin de raccourci pour gagner du temps

**Nouveau prompt** :

- Pattern de code utilisé régulièrement
- Convention spécifique au projet
- Template réutilisable

### Versionner avec Git

```bash
# ✅ Committé pour l'équipe
git add .claude/
git commit -m "feat: add feature workflow command"

# ✅ Ignoré pour config locale
echo "settings.local.json" >> .gitignore
```

## Comparaison : agents vs commands vs prompts

| Critère                    | Agents                     | Commands                   | Prompts                        |
| -------------------------- | -------------------------- | -------------------------- | ------------------------------ |
| **Complexité**             | Élevée, multi-étapes       | Moyenne, workflow linéaire | Faible, template simple        |
| **Durée d'exécution**      | Longue (plusieurs minutes) | Moyenne (quelques minutes) | Courte (instantané)            |
| **Logique conditionnelle** | Oui                        | Limitée                    | Non                            |
| **Réutilisabilité**        | Haute, contexte spécifique | Haute, workflow générique  | Très haute, structure pure     |
| **Exemple**                | Clean Architecture         | /feature, /refactor        | component.md, server-action.md |

## Résumé

Le dossier `.claude` organise la configuration de Claude Code en 5 catégories principales :

1. **agents/** - Workflows complexes et spécialisés (à la demande)
2. **commands/** - Slash commands pour workflows récurrents (à la demande)
3. **prompts/** - Templates de code et structures (à la demande)
4. **hooks/** - Documentation sur l'automatisation (documentation)
5. **docs/** - Guides et documentation (documentation)
6. **settings.local.json** - Configuration principale (au démarrage)

Cette structure permet de :

- ✅ Standardiser les workflows d'équipe
- ✅ Réutiliser les patterns éprouvés
- ✅ Accélérer le développement
- ✅ Maintenir la cohérence du code
- ✅ Onboarder rapidement de nouveaux développeurs
