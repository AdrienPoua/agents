## 1. Vue d'ensemble

### Qu'est-ce que le dossier `.cursor` ?

Le dossier `.cursor` est un système de configuration pour **Cursor IDE** et **Claude Code** qui permet de personnaliser le comportement des assistants IA en fonction de votre projet. Il fournit :

- **Règles de développement** : Conventions de code, patterns architecturaux
- **Contexte projet** : Technologies utilisées, structure du code
- **Instructions spécifiques** : Comment l'IA doit générer du code

### Pourquoi l'utiliser ?

✅ **Cohérence** : Code uniforme selon vos conventions
✅ **Productivité** : L'IA génère du code conforme dès le départ
✅ **Collaboration** : Toute l'équipe partage les mêmes règles
✅ **Qualité** : Réduit les erreurs et le temps de review
✅ **Spécificité** : Adapté à votre architecture (Clean Architecture dans notre cas)

### Compatibilité

| Outil              | Support `.cursor/` | Notes                          |
| ------------------ | ------------------ | ------------------------------ |
| **Cursor IDE**     | ✅ Natif           | Support complet                |
| **Claude Code**    | ✅ Supporté        | Lit automatiquement les règles |
| **GitHub Copilot** | ❌ Non             | Système propriétaire           |
| **VS Code**        | ⚠️ Partiel         | Dépend des extensions          |

## 3. Format des fichiers `.mdc`

### Qu'est-ce qu'un fichier `.mdc` ?

**MDC** = **M**ark**D**own **C**ursor

C'est un format Markdown avec métadonnées YAML pour définir des règles contextuelles.

### Structure d'un fichier `.mdc`

```markdown
---
title: 'Nom de la règle'
id: unique-identifier
description: "Description courte (affichée dans l'UI)"
priority: 100
globs:
  - '**/*.ts'
  - '**/*.tsx'
alwaysApply: false
---

# Contenu de la règle

Instructions pour l'IA en Markdown...
```

### Champs du front matter YAML

#### `title` (optionnel)

**Type** : `string`
**Usage** : Nom affiché dans l'interface Cursor

```yaml
title: 'Next.js App Router Conventions'
```

#### `id` (optionnel)

**Type** : `string`
**Usage** : Identifiant unique pour référencement

```yaml
id: nextjs-app-router
```

#### `description` (recommandé)

**Type** : `string`
**Usage** : Description courte de la règle (100 caractères max)

```yaml
description: 'Conventions Next.js 14 avec App Router et TypeScript'
```

#### `priority` (optionnel)

**Type** : `number`
**Usage** : Ordre d'application (plus haut = prioritaire)
**Défaut** : 0

```yaml
priority: 100 # Haute priorité
```

#### `globs` (optionnel)

**Type** : `string` ou `array`
**Usage** : Patterns de fichiers pour activation automatique

```yaml
# Un seul pattern
globs: "**/*.tsx"

# Plusieurs patterns
globs:
  - "**/*.ts"
  - "**/*.tsx"
  - "src/app/**/*"
```

#### `alwaysApply` (optionnel)

**Type** : `boolean`
**Usage** : Appliquer automatiquement à chaque requête
**Défaut** : `false`

```yaml
alwaysApply: true # Règle Always
```

### Exemple complet : notre fichier actuel

```markdown
---
description: 'Next.js 14 best practices with Clean Architecture'
globs:
  - '**/*.ts'
  - '**/*.tsx'
---

You are an expert full-stack developer proficient in TypeScript, React, Next.js...

### Objective

- Create a Next.js solution that is not only functional but also adheres to best practices...

### Code Style and Structure

- Write concise, technical TypeScript code with accurate examples...
```

## 4. Types de règles

Cursor supporte 4 types de règles avec différents comportements d'activation.

### 1. Always Rules 🌐

**Comportement** : Appliquées à **chaque requête** automatiquement

**Configuration** :

```yaml
---
alwaysApply: true
---
```

**Cas d'usage** :

- Standards de code globaux
- Principes architecturaux (Clean Architecture)
- Conventions de nommage
- Gestion d'erreurs

**Exemple** :

```markdown
---
title: 'Clean Architecture Principles'
description: 'Core architectural patterns for the project'
alwaysApply: true
priority: 100
---

# Architecture Rules

This project follows Clean Architecture with strict layer separation:

1. **Domain** → Core business logic, no dependencies
2. **Application** → Use cases
3. **Infrastructure** → External implementations
4. **Presentation** → UI layer

**Rule**: Always respect dependency direction: UI → Presentation → Application → Domain ← Infrastructure
```

### 2. Auto-Attach Rules 📎

**Comportement** : Activées automatiquement quand les fichiers correspondent aux patterns `globs`

**Configuration** :

```yaml
---
globs:
  - '**/*.tsx'
  - 'src/components/**/*'
---
```

**Cas d'usage** :

- Conventions spécifiques à un type de fichier
- Règles par technologie (React, TypeScript)
- Standards par dossier (components, hooks, actions)

**Exemple** :

```markdown
---
title: 'React Component Conventions'
description: 'Standards for React components'
globs:
  - 'src/components/**/*.tsx'
  - 'src/app/**/*.tsx'
---

# React Component Rules

## Structure

- Use functional components only
- Favor Server Components over Client Components
- Add 'use client' directive only when necessary

## Naming

- PascalCase for component files: UserProfile.tsx
- Lowercase with dashes for directories: user-profile/

## Props

- Define TypeScript interface for props
- Use descriptive prop names
- Avoid prop drilling (max 2 levels)
```

### 3. Agent-Requested Rules 🤖

**Comportement** : L'IA décide de les appliquer selon le contexte

**Configuration** :

```yaml
---
title: 'Security Best Practices'
description: 'Security guidelines - applied when AI detects security context'
# Pas de alwaysApply ni globs
---
```

**Cas d'usage** :

- Sécurité
- Performance
- Optimisations
- Patterns avancés

**Exemple** :

```markdown
---
title: 'Security Guidelines'
description: 'Security best practices for sensitive operations'
---

# Security Rules

Apply these rules when handling:

- User authentication
- Data validation
- Environment variables
- API endpoints

## Authentication

- Always use Supabase Auth
- Never expose API keys in client code
- Validate session server-side

## Data Validation

- Use Zod schemas for all inputs
- Sanitize user inputs
- Implement rate limiting
```

### 4. Manual Rules 📝

**Comportement** : Appliquées uniquement sur demande explicite de l'utilisateur

**Configuration** :

```yaml
---
title: 'Component Generator'
description: 'Template for generating new components'
# Aucune activation automatique
---
```

**Cas d'usage** :

- Templates de génération
- Workflows spécifiques
- Migrations complexes

**Exemple** :

```markdown
---
title: 'New Entity Workflow'
description: 'Steps to create a new Clean Architecture entity'
---

# Create New Entity Workflow

Follow these steps to add a new entity (e.g., "Player"):

## 1. Domain Layer

- [ ] Create `core/domain/entities/player.entity.ts`
- [ ] Define interface and class with getters
- [ ] Add `toObject()` method

## 2. DTOs

- [ ] Create `core/domain/dtos/player.dto.ts`
- [ ] Create `core/application/dtos/player.dto.ts`
- [ ] Create `core/infrastructure/supabase/dtos/player.dto.ts`

## 3. Repository

- [ ] Create interface `core/domain/repositories/player.repository.ts`
- [ ] Implement `core/infrastructure/supabase/repositories/player.repository.ts`
- [ ] Add to `RepositoryFactory`

## 4. Mappers

- [ ] Create `core/infrastructure/supabase/mappers/player.mapper.ts`
- [ ] Implement `toDomain()` and `toPersistence()`

## 5. Use Cases

- [ ] Create in `core/application/usecases/Player/`
- [ ] Implement: GetAll, GetById, Create, Update, Delete

## 6. Presentation

- [ ] Create Server Actions in `core/presentation/actions/players/`
- [ ] Create React Query hooks in `core/presentation/hooks/players/`

## 7. UI

- [ ] Create components in `src/components/`
- [ ] Create pages in `src/app/`
```

### Comparaison des types

| Type                | Activation                   | Usage principal            | Configuration         |
| ------------------- | ---------------------------- | -------------------------- | --------------------- |
| **Always**          | Automatique (toujours)       | Standards globaux          | `alwaysApply: true`   |
| **Auto-Attach**     | Automatique (si globs match) | Règles par fichier/dossier | `globs: ["**/*.tsx"]` |
| **Agent-Requested** | Automatique (IA décide)      | Contextes spécifiques      | Aucun flag            |
| **Manual**          | Manuelle (utilisateur)       | Templates, workflows       | Aucun flag            |

## 5. Patterns Glob

### Syntaxe glob

Les patterns glob permettent de cibler des fichiers spécifiques pour les règles **Auto-Attach**.

### Syntaxe de base

| Pattern | Description                          | Exemple                                |
| ------- | ------------------------------------ | -------------------------------------- |
| `*`     | N'importe quel caractère (sauf `/`)  | `*.ts` → tous les fichiers `.ts`       |
| `**`    | N'importe quel répertoire (récursif) | `**/*.tsx` → tous les `.tsx` partout   |
| `?`     | Un seul caractère                    | `page?.tsx` → `page1.tsx`, `pageA.tsx` |
| `[abc]` | Caractère parmi les options          | `[pt]age.tsx` → `page.tsx`, `tage.tsx` |
| `{a,b}` | Alternative                          | `*.{ts,tsx}` → `.ts` ou `.tsx`         |

### Exemples concrets pour notre projet

#### Cibler tous les fichiers TypeScript

```yaml
globs:
  - '**/*.ts'
  - '**/*.tsx'
```

#### Cibler uniquement les composants React

```yaml
globs:
  - 'src/components/**/*.tsx'
  - 'src/app/**/*.tsx'
```

#### Cibler les Server Actions

```yaml
globs:
  - 'core/presentation/actions/**/*.ts'
```

#### Cibler les Use Cases

```yaml
globs:
  - 'core/application/usecases/**/*.ts'
```

#### Cibler les repositories

```yaml
globs:
  - 'core/infrastructure/supabase/repositories/**/*.ts'
```

#### Cibler les pages Next.js

```yaml
globs:
  - 'src/app/**/page.tsx'
  - 'src/app/**/layout.tsx'
```

#### Cibler les hooks React Query

```yaml
globs:
  - 'core/presentation/hooks/**/*.ts'
```

#### Cibler les fichiers de test

```yaml
globs:
  - '**/*.test.ts'
  - '**/*.test.tsx'
  - '**/*.spec.ts'
```

#### Cibler une catégorie spécifique

```yaml
globs:
  - 'src/app/(dashboard)/**/*.tsx' # Uniquement le dashboard
```

#### Exclure des fichiers (avec négation)

```yaml
globs:
  - '**/*.ts'
  - '!**/*.test.ts' # Exclure les tests
  - '!**/node_modules/**' # Exclure node_modules
```

### Cas d'usage avancés

#### Règle pour les entités du domaine

```markdown
---
title: 'Domain Entity Pattern'
description: 'Entity structure with getters and immutability'
globs:
  - 'core/domain/entities/**/*.ts'
---

# Domain Entity Rules

All entities must follow this pattern:

1. **Interface** + **Class** implementing it
2. **Private readonly** fields with `_` prefix
3. **Public getters** for all fields
4. **toObject()** method for serialization
5. **No setters** (immutability)

Example:
\`\`\`typescript
export interface Team {
id: string
name: string
}

export class TeamEntity implements Team {
private readonly \_id: string
private readonly \_name: string

constructor(data: Team) {
this.\_id = data.id
this.\_name = data.name
}

get id(): string { return this.\_id }
get name(): string { return this.\_name }

toObject(): Team {
return { id: this.id, name: this.name }
}
}
\`\`\`
```

#### Règle pour les formulaires

```markdown
---
title: 'Form Validation with Zod'
description: 'Standards for form handling with React Hook Form + Zod'
globs:
  - 'src/components/**/*form*.tsx'
  - 'src/app/**/*form*.tsx'
---

# Form Rules

All forms must use:

- **React Hook Form** for form state
- **Zod** for validation schema
- **Server Actions** for submission

Pattern:
\`\`\`typescript
'use client'

import { useForm } from 'react-hook-form'
import { zodResolver } from '@hookform/resolvers/zod'
import { z } from 'zod'

const schema = z.object({
name: z.string().min(1, 'Requis'),
email: z.string().email('Email invalide'),
})

type FormData = z.infer<typeof schema>

export function MyForm() {
const form = useForm<FormData>({
resolver: zodResolver(schema),
})

async function onSubmit(data: FormData) {
await createAction(data)
}

return <form onSubmit={form.handleSubmit(onSubmit)}>...</form>
}
\`\`\`
```

## 6. Exemples de règles pour notre projet

Voici des exemples de règles `.mdc` adaptées à notre architecture Clean Architecture et notre stack technique.

### 6.1 Architecture globale (Always Rule)

**Fichier** : `.cursor/rules/architecture.mdc`

```markdown
---
title: 'Clean Architecture Principles'
description: 'Core architectural patterns for Argenteuil Basketball'
alwaysApply: true
priority: 100
---

# Clean Architecture Rules

This project follows **Clean Architecture** with strict layer separation.

## Layer Structure

\`\`\`
core/
├── domain/ # Business logic (no dependencies)
├── application/ # Use cases
├── infrastructure/ # External implementations
├── presentation/ # Server Actions + React Query hooks
└── shared/ # Utilities

src/ # UI layer
├── app/ # Next.js App Router
└── components/ # React components
\`\`\`

## Dependency Rule

**Direction** : `UI → Presentation → Application → Domain ← Infrastructure`

- Domain has **no external dependencies**
- Infrastructure **implements** domain interfaces
- Presentation **orchestrates** use cases
- UI **consumes** presentation layer

## Adding New Features

Follow this order:

1. Domain (entities, repository interfaces)
2. Infrastructure (repository implementations, mappers)
3. Application (use cases)
4. Presentation (server actions, hooks)
5. UI (components, pages)

## Repository Pattern

- All data access through repositories
- Use `RepositoryFactory` for instantiation
- Client type: `'browser'` for server actions

## Use Cases

- One class per business operation
- Implement `BaseUseCase<TInput, TOutput>`
- Handle errors with `ErrorHandler`

## DTOs and Mappers

Three layers:

- **Database DTO** (Supabase format)
- **Domain Entity** (business object)
- **Application DTO** (API/UI format)

Mappers transform between layers.
```

### 6.2 TypeScript strict (Auto-Attach)

**Fichier** : `.cursor/rules/typescript.mdc`

```markdown
---
title: 'TypeScript Strict Mode'
description: 'TypeScript conventions and strict type safety'
globs:
  - '**/*.ts'
  - '**/*.tsx'
---

# TypeScript Rules

## Strict Mode

- **Always** use TypeScript strict mode
- **No** `any` types (use `unknown` if needed)
- **No** `as` assertions (use type guards)
- **No** `@ts-ignore` (fix the issue)

## Type Definitions

### Interfaces vs Types

Use **interfaces** for:

- Object shapes
- Component props
- Entity definitions

Use **types** for:

- Unions and intersections
- Mapped types
- Utility types

### Naming Conventions

- **Interfaces**: PascalCase, no `I` prefix
  - ✅ `UserProfile`
  - ❌ `IUserProfile`

- **Types**: PascalCase with context
  - ✅ `UserRole`, `CreateUserDTO`

- **Enums**: PascalCase (prefer const objects or unions)
  - ✅ `const UserRole = { ADMIN: 'admin', USER: 'user' } as const`

### Type Guards

Use type guards for runtime checks:

\`\`\`typescript
function isError(value: unknown): value is Error {
return value instanceof Error
}

if (isError(error)) {
console.error(error.message) // Type-safe
}
\`\`\`

### Generic Constraints

Use constraints for better type inference:

\`\`\`typescript
// ❌ Bad
function getProperty<T>(obj: T, key: string) {
return obj[key] // Error
}

// ✅ Good
function getProperty<T extends object, K extends keyof T>(obj: T, key: K) {
return obj[key] // Type-safe
}
\`\`\`

## Function Types

Prefer function declarations:

\`\`\`typescript
// ✅ Preferred
function createUser(input: CreateUserDTO): Promise<User> {
// ...
}

// ⚠️ Use for callbacks/higher-order functions
const createUser = async (input: CreateUserDTO): Promise<User> => {
// ...
}
\`\`\`

## Import/Export

Use named exports (except for page.tsx, layout.tsx):

\`\`\`typescript
// ✅ Good
export function UserProfile() {}
export type UserProfileProps = {}

// ❌ Avoid
export default function UserProfile() {}
```

### 6.3 React/Next.js (Auto-Attach)

**Fichier** : `.cursor/rules/nextjs.mdc` (version améliorée de l'existant)

```markdown
---
title: 'Next.js 14 App Router Conventions'
description: 'Next.js, React, and Server Components best practices'
globs:
  - 'src/**/*.tsx'
  - 'src/**/*.ts'
---

# Next.js & React Rules

## Component Strategy

### Server Components (Default)

Use **Server Components** by default:

\`\`\`typescript
// src/app/teams/page.tsx
export default async function TeamsPage() {
const teams = await readTeams() // Direct server action call
return <TeamList teams={teams.map(t => t.toObject())} />
}
\`\`\`

Benefits:

- Direct data fetching
- No client-side JavaScript
- Better performance
- SEO-friendly

### Client Components

Add `'use client'` **only** when needed:

- Interactive state (`useState`, `useReducer`)
- React hooks (`useEffect`, `useRef`)
- Event handlers (onClick, onChange)
- Browser APIs (localStorage, window)
- React Query hooks

\`\`\`typescript
'use client'

import { useState } from 'react'

export function Counter() {
const [count, setCount] = useState(0)
return <button onClick={() => setCount(c => c + 1)}>{count}</button>
}
\`\`\`

## File Structure

### App Router Conventions

\`\`\`
src/app/
├── (site)/ # Public site
│ ├── layout.tsx # Root layout
│ ├── page.tsx # Homepage
│ └── teams/
│ ├── page.tsx # /teams
│ └── [id]/
│ └── page.tsx # /teams/:id
├── (dashboard)/ # Admin dashboard

### Special Files

- `page.tsx` - Page component (default export)
- `layout.tsx` - Layout wrapper (default export)
- `loading.tsx` - Loading state (default export)
- `error.tsx` - Error boundary (use client, default export)
```

```markdown
      // 2. Repository call
```

```markdown
\`\`\`typescript
```

## 10. Configuration MCP

### Qu'est-ce que MCP ?

**MCP** (Model Context Protocol) permet aux assistants IA d'accéder à des contextes externes :

- Documentation de bibliothèques (Context7)
- Bases de données
- APIs
- Systèmes de fichiers

### MCP dans notre projet

Nous utilisons **Context7** pour accéder à la documentation à jour des bibliothèques.

### Configuration

**Fichier** : `.cursor/mcp.json` (à créer si nécessaire)

```json
{
  "mcpServers": {
    "context7": {
      "command": "npx",
      "args": ["-y", "@context7/mcp-server"],
      "env": {
        "CONTEXT7_API_KEY": "${CONTEXT7_API_KEY}"
      }
    }
  }
}
```

### Utilisation

Une fois configuré, l'IA peut accéder à la documentation :

```
User: Comment utiliser React Query avec suspense ?
AI: [Fetches latest React Query docs via Context7]
    Voici comment utiliser suspense avec React Query v5...
```

### Autres MCP servers utiles

#### Supabase

```json
{
  "supabase": {
    "command": "npx",
    "args": ["-y", "@supabase/mcp-server"],
    "env": {
      "SUPABASE_URL": "${NEXT_PUBLIC_SUPABASE_URL}",
      "SUPABASE_KEY": "${SUPABASE_SERVICE_ROLE_KEY}"
    }
  }
}
```

#### PostgreSQL

```json
{
  "postgres": {
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-postgres"],
    "env": {
      "DATABASE_URL": "${DATABASE_URL}"
    }
  }
}
```

### Sécurité

⚠️ **Attention** : Ne jamais hardcoder les secrets dans `.cursor/mcp.json`

✅ **Bonne pratique** : Utiliser des variables d'environnement

```json
{
  "env": {
    "API_KEY": "${MY_API_KEY}" // Lit depuis .env.local
  }
}
```

## 11. Maintenance et mise à jour

### Quand mettre à jour les règles ?

#### 1. Nouveau pattern adopté

Vous adoptez un nouveau pattern dans le projet :

```markdown
// Ajout d'une règle
.cursor/rules/optimistic-updates.mdc
```

#### 2. Changement de stack

Migration ou ajout de technologie :

```markdown
// Avant : Jest
.cursor/rules/jest.mdc

// Après : Vitest
.cursor/rules/vitest.mdc
```

#### 3. Feedback de l'équipe

L'équipe identifie des erreurs récurrentes :

```markdown
// Nouvelle règle pour prévenir
.cursor/rules/common-mistakes.mdc
```

#### 4. Évolution des conventions

Les conventions évoluent avec le projet :

```markdown
// Mise à jour
.cursor/rules/naming.mdc (v2)
```

#### 5. Mise à jour des dépendances

Framework ou librairie mise à jour :

```markdown
// Next.js 14 → 15
.cursor/rules/nextjs.mdc (updated for v15)
```

### Processus de mise à jour

#### Étape 1 : Identifier le besoin

- Code reviews récurrentes sur le même sujet
- Nouveau pattern dans le projet
- Feedback de l'IA (code non conforme)

#### Étape 2 : Modifier la règle

```bash
# Ouvrir le fichier
code .cursor/rules/nextjs.mdc

# Ajouter/modifier la section
```

#### Étape 3 : Ajouter la date de modification

```yaml
---
title: 'Next.js Conventions'
updated: '2025-10-30'
---
```

#### Étape 4 : Tester

1. Ouvrir un fichier concerné
2. Demander à l'IA de générer du code
3. Vérifier la conformité

#### Étape 5 : Commiter

```bash
git add .cursor/rules/nextjs.mdc
git commit -m "docs(cursor): update Next.js rules for App Router"
git push
```

#### Étape 6 : Communiquer

Informer l'équipe :

- Message Slack/Teams
- Pull Request description
- Documentation changelog

### Versionnement

#### Approche 1 : Date dans front matter

```yaml
---
title: 'TypeScript Rules'
version: '1.2.0'
updated: '2025-10-30'
---
```

#### Approche 2 : Changelog dans le fichier

```markdown
---
title: 'Next.js Rules'
---

# Next.js Conventions

## Changelog

### 2025-10-30 - v1.2.0

- Added Server Actions pattern
- Updated for Next.js 15

### 2025-09-15 - v1.1.0

- Added metadata generation rules

### 2025-08-01 - v1.0.0

- Initial version
```

#### Approche 3 : Git history

```bash
# Voir l'historique d'une règle
git log -p .cursor/rules/nextjs.mdc
```

### Nettoyage régulier

#### Supprimer les règles obsolètes

```bash
# Règle plus pertinente
git rm .cursor/rules/old-pattern.mdc
git commit -m "chore(cursor): remove obsolete pattern rule"
```

#### Fusionner les règles similaires

```bash
# Avant
.cursor/rules/react-components.mdc
.cursor/rules/react-hooks.mdc

# Après
.cursor/rules/react.mdc (fusionné)
```

### Audit périodique

**Fréquence recommandée** : Trimestriel

**Checklist d'audit** :

- [ ] Règles toujours pertinentes ?
- [ ] Exemples de code à jour ?
- [ ] Pas de contradictions ?
- [ ] Globs optimisés ?
- [ ] Pas de règles redondantes ?
- [ ] Cohérence avec CLAUDE.md ?
- [ ] Tests avec l'IA concluants ?

### Workflow de mise à jour en équipe

#### 1. Proposition

Créer une Pull Request :

```
feat(cursor): add React Query optimistic updates rule

- Add new rule for optimistic UI patterns
- Include examples with TanStack Query
- Target hooks directory with globs
```

#### 2. Review

L'équipe review la règle :

- Clarté des instructions
- Pertinence des exemples
- Cohérence avec l'existant

#### 3. Test

Reviewer teste avec l'IA :

```
"Generate a hook for updating a team with optimistic update"
```

#### 4. Merge

Une fois validée, merger et déployer.

### Migration `.cursorrules` → `.cursor/rules/`

Si vous avez un ancien fichier `.cursorrules` :

#### Étape 1 : Analyser le contenu

```bash
cat .cursorrules
```

#### Étape 2 : Découper par contexte

```
.cursorrules (1000 lignes)
↓
.cursor/rules/architecture.mdc (200 lignes)
.cursor/rules/nextjs.mdc (300 lignes)
.cursor/rules/typescript.mdc (200 lignes)
.cursor/rules/testing.mdc (150 lignes)
```

#### Étape 3 : Ajouter front matter

```yaml
---
title: 'Architecture Rules'
description: 'Clean Architecture patterns'
alwaysApply: true
---
```

#### Étape 4 : Ajouter globs pour Auto-Attach

```yaml
---
globs:
  - '**/*.tsx'
---
```

#### Étape 5 : Tester et supprimer `.cursorrules`

```bash
# Tester les nouvelles règles
# ...

# Supprimer l'ancien fichier
git rm .cursorrules
git commit -m "refactor(cursor): migrate to .cursor/rules/ structure"
```

## Conclusion

Le dossier `.cursor` est un outil puissant pour :

✅ **Améliorer la qualité** du code généré par l'IA
✅ **Standardiser** les pratiques au sein de l'équipe
✅ **Accélérer** le développement avec des conventions claires
✅ **Réduire** les erreurs et le temps de code review
✅ **Adapter** l'IA à votre architecture (Clean Architecture)

### Points clés à retenir

1. **Structure** : `.cursor/rules/` pour les règles, `.cursor/mcp/` pour les serveurs MCP
2. **Format** : Fichiers `.mdc` avec front matter YAML + contenu Markdown
3. **Types de règles** : Always, Auto-Attach, Agent-Requested, Manual
4. **Globs** : Ciblez précisément avec des patterns glob
5. **Complémentarité** : `.cursor/` pour les règles de code, `CLAUDE.md` pour le contexte
6. **Bonnes pratiques** : Concis, spécifique, avec exemples, versionné
7. **Maintenance** : Audit régulier, mise à jour, nettoyage
