# Template — Document post-signature (cahier des charges projet)

> Ce document est produit APRÈS signature du devis et versement de l'acompte.
> C'est le livrable de la phase de cadrage — la première phase payante du projet.
>
> **Objectif** : Être la référence unique du projet. Quand il y a un doute sur le périmètre, le planning, ou un livrable, on ouvre ce document.
>
> **Qui le produit** : Moi, après le RDV de cadrage. Le client le valide. Une fois validé, c'est le contrat moral du "ce qu'on construit".
>
> **Longueur cible** : 6-10 pages. Assez détaillé pour ne laisser aucune ambiguïté, assez concis pour être relu.
>
> **Relation avec le Template 1** : Ce document remplace et précise le périmètre indicatif du document de mise en relation. Toute divergence entre les deux est normale (le cadrage affine les choses) — ce document fait foi.

---

## Section 1 — Contexte & objectifs

**But** : Poser les fondations. Tout le monde est aligné sur le "pourquoi" avant de parler du "quoi".

### Contenu

- Rappel du projet en 3-4 phrases (repris du Template 1, affiné après le cadrage)
- **Objectif principal du site** en 1 phrase claire et mesurable
- **Cible** : qui va visiter ce site ? Quel profil, quel niveau de connaissance, quelle intention ?
- **Action attendue** : que doit faire le visiteur ? (prise de contact, achat, inscription, etc.)
- Éléments de contexte issus du cadrage (deadline business, événement, contrainte particulière)
- **Référence** : _"Ce document remplace et précise le périmètre indicatif de la proposition de collaboration du [date du Template 1]."_

### Règles

- L'objectif doit être **spécifique**. Pas "faire connaître l'entreprise" mais "provoquer une prise de contact de la part de décideurs industriels".
- La cible doit être **concrète**. Pas "tout le monde" mais "directeurs innovation de groupes industriels, francophones et anglophones".
- Si le client a plusieurs cibles, les hiérarchiser : primaire, secondaire.

### Exemple

> **Projet** : Site vitrine bilingue FR/EN pour YETI.live, wearable connecté deep-tech en recherche de partenaire industriel.
>
> **Objectif** : Générer des prises de contact qualifiées de la part de partenaires industriels potentiels.
>
> **Cible primaire** : Décideurs industriels (directeurs innovation, VP partenariats, CTO) de groupes capables de produire du hardware en série.
>
> **Cible secondaire** : Investisseurs et fonds spécialisés sport-tech / health-tech.
>
> **Action attendue** : Remplir le formulaire de contact ou envoyer un email.
>
> **Contexte** : Pas de deadline stricte. Le projet sert d'outil de prospection permanent.
>
> _Ce document remplace et précise le périmètre indicatif de la proposition de collaboration du 27 février 2026._

## Section 2 — Architecture du site

**But** : Définir la structure exacte de ce qu'on construit. Section par section, dans l'ordre.

### Contenu

Pour chaque section/page du site :
- **Nom de la section**
- **Rôle** : à quelle question du visiteur cette section répond (1 phrase)
- **Contenu prévu** : ce qu'on y met (bullet points)
- **CTA** (si applicable) : quelle action on pousse

### Règles

- Chaque section doit répondre à UNE question du visiteur. Si elle répond à deux questions, c'est deux sections.
- L'ordre des sections = l'ordre narratif. Le site se déroule comme un argumentaire.
- Pour un site one-page : lister les sections dans l'ordre du scroll.
- Pour un site multi-pages : faire un sitemap + détailler chaque page.

### Exemple (site one-page, 8 sections)

> **1. Hero**
> - Rôle : Capter l'attention en 3 secondes. Dire ce que c'est et pourquoi c'est différent.
> - Contenu : Headline, sous-titre, visuel produit, CTA "Découvrir" ou "Nous contacter"
>
> **2. Le Problème**
> - Rôle : Créer l'identification. "On comprend votre marché."
> - Contenu : Le constat du gap dans le marché (tracking physique uniquement, pas de body-mind)
>
> **3. La Solution**
> - Rôle : Présenter le produit et ce qu'il fait.
> - Contenu : Le produit (1 capteur thoracique), les métriques, la différenciation somatopsychique
>
> _[...etc pour chaque section]_

## Section 3 — Direction artistique

**But** : Définir l'identité visuelle du projet. C'est ici que va tout ce qui était volontairement exclu du Template 1.

### Processus de collecte

Avant de rédiger cette section, un échange dédié sur les préférences visuelles est nécessaire. Le processus :

1. **Questionnaire amont** (envoyé par email avant le RDV) : 5 questions clés — "Quels sites vous inspirent ?", "Quels sites vous repoussent ?", "3 mots pour décrire l'ambiance souhaitée", "Avez-vous une charte graphique existante ?", "Y a-t-il des couleurs imposées/interdites ?"
2. **Échange DA** (15-20 min pendant le RDV de cadrage ou en call dédié) : on passe en revue les réponses, je montre 3-5 références visuelles, on aligne les attentes.
3. **Synthèse ci-dessous** : je formalise l'accord visuel.

### Contenu

- **Intention globale** : 2-3 mots-clés qui définissent l'ambiance (ex: "premium, scientifique, sobre")
- **Palette de couleurs** : couleurs principales, secondaires, neutres (codes hex)
- **Typographie** : font titres, font corps, hiérarchie
- **Imagerie** : style des visuels (photo, illustration, 3D, abstrait), traitement (sombre, lumineux, contrasté)
- **Références visuelles** : 3-5 sites/marques qui illustrent la direction (liens)
- **Ce qu'on évite** : anti-références, choses que le client n'aime pas

### Règles

- Toujours inclure des références visuelles concrètes — les mots ne suffisent pas pour aligner les attentes.
- Les anti-références sont aussi importantes que les références. "Pas de flat design coloré type startup" est une info précieuse.
- Si un moodboard ou une preview HTML a été produit, le lier ici.

### Exemple

> **Intention** : Premium, scientifique, sobre. La dualité froid (précision, tech) / chaud (humain, vivant).
>
> **Palette** :
> - Principal : `#0A0F1C` (fond sombre profond)
> - Accent chaud : `#E8734A` (énergie, vie)
> - Accent froid : `#4A9BE8` (tech, précision)
> - Texte : `#F0F0F0` (clair sur sombre)
>
> **Typographie** : Inter (corps), Instrument Serif (titres)
>
> **Références** : [lien 1], [lien 2], [lien 3]
>
> **On évite** : templates SaaS génériques, flat design coloré, stock photos sportives clichés.

## Section 4 — Spécifications techniques

**But** : Cadrer les choix techniques. C'est la section de référence pour le développement.

### Contenu

- **Stack** : framework, langage, CSS, hébergement
- **Responsive** : breakpoints ciblés (mobile, tablette, desktop)
- **Performance** : objectifs Lighthouse (performance, accessibilité, SEO, best practices)
- **SEO** : balises meta, structure sémantique, sitemap, robots.txt, Open Graph
- **Accessibilité** : niveau visé (RGAA, WCAG AA), points clés (contrastes, navigation clavier, ARIA, alt texts)
- **Bilingue** (si applicable) : mécanisme de switch, gestion des traductions
- **Formulaire** : champs, validation, destination des messages, conformité RGPD (mentions légales, consentement)
- **Analytics** : outil, événements trackés, conformité RGPD
- **Nom de domaine** : qui l'achète, quel registrar, configuration DNS
- **Hébergement** : plateforme, plan, coût mensuel

### Règles

- Être précis sur les objectifs mesurables (Lighthouse 90+, WCAG AA, etc.) — c'est ce qui définit "terminé" ou "pas terminé".
- Mentionner les choix techniques avec 1 phrase de justification si le client n'est pas technique : _"Next.js — framework standard du marché, performant, facilement reprenable par un autre développeur."_
- Séparer ce qui est à la charge du client (nom de domaine, contenu) de ce qui est à ma charge.
- **Analytics est standard, pas optionnel.** Livrer un site sans aucun suivi de fréquentation, c'est livrer un produit incomplet. Au minimum : Plausible, Umami ou Vercel Analytics (léger, RGPD-friendly, pas de bandeau cookies). Seule exception : le client demande explicitement de ne pas en mettre.

### Exemple

> **Stack** : Next.js 15 + TypeScript + Tailwind CSS — stack moderne, standard, maintenable.
>
> **Hébergement** : Vercel (gratuit pour ce type de site, performant, CDN mondial).
>
> **Performance** : Lighthouse 90+ sur les 4 catégories.
>
> **Accessibilité** : WCAG AA — contrastes, navigation clavier, textes alternatifs, structure sémantique.
>
> **Bilingue** : FR/EN, switch intégré dans le header, traductions gérées côté code (pas de traduction automatique).
>
> **Formulaire** : Nom, email, message, consentement RGPD. Emails envoyés à [adresse client]. Mentions légales et politique de confidentialité incluses.
>
> **Analytics** : Vercel Analytics (inclus dans l'hébergement, conforme RGPD, pas de bandeau cookies nécessaire). Métriques suivies : visites, pages vues, sources de trafic, appareil. Événement personnalisé : soumission du formulaire de contact.
>
> **Nom de domaine** : À acheter par le client (~10-15€/an). Je configure le DNS et le déploiement.

## Section 6 — Planning détaillé

**But** : Calendrier précis avec des dates, des jalons et des points de validation.

### Contenu

Un planning phase par phase avec :
- **Phase** : nom
- **Dates** : début → fin
- **Livrable de la phase** : ce que je produis
- **Validation client** : ce que le client doit valider pour passer à la suite
- **Dépendance** : ce qui doit être fourni/fait avant de commencer

### Format type

| Phase | Dates | Livrable | Validation client | Dépendance |
|-------|-------|----------|-------------------|------------|
| Cadrage | S1 | Ce document | Validation du cahier des charges | Acompte versé |
| Direction artistique | S1-S2 | Maquettes desktop + mobile | Choix de la direction (tour 1/3) | Cahier des charges validé |
| Contenu | S2-S3 | Textes FR finalisés | Validation des textes | DA validée |
| Développement | S3-S4 | Site fonctionnel en staging | — | Textes + DA validés |
| Recette | S4 | Corrections / ajustements | Tests + retours | Site en staging |
| Traduction | S4-S5 | Version EN | Validation traduction | Textes FR validés |
| Mise en ligne | S5 | Site en production | Validation finale | Recette terminée |

### Règles

- **Chaque phase a une condition de sortie claire** (validation client). Pas de phase qui "traîne".
- Mentionner les **points de validation** : ce sont les moments où le projet peut s'arrêter si le client ne répond pas.
- Le planning est un engagement mutuel : je m'engage sur les délais SI le client respecte les siens.
- Inclure une marge de sécurité (si je pense livrer en 3 semaines, mettre 4).

### Clauses de délai à inclure systématiquement

> _"Ce planning est un engagement mutuel. Les dates de livraison sont conditionnées au respect des délais de validation et de fourniture de contenu côté client."_
>
> _"Délai de validation : [X] jours ouvrés maximum par jalon. Passé ce délai sans retour, la version est considérée comme validée et la phase suivante démarre. Tout retard de validation supérieur à [Y] jours ouvrés peut entraîner une reprogrammation du projet selon ma disponibilité."_

## Section 7 — Processus de révision

**But** : Cadrer les révisions pour éviter le scope creep et les allers-retours infinis.

### Contenu

- Nombre de tours de révision inclus par phase
- Ce qui compte comme une "révision"
- Ce qui ne compte PAS comme une révision (et qui est un changement de périmètre)
- Le processus : comment le client fait ses retours, dans quel format, dans quel délai

### Règles

| Type de retour | Compte comme révision ? |
|---------------|------------------------|
| "Change la couleur du bouton" | Non — ajustement mineur |
| "Déplace cette section et réécris le texte" | Oui — 1 tour |
| "Ajoute une nouvelle section qu'on n'avait pas prévue" | Non — changement de périmètre → devis complémentaire |
| "Le site ne s'affiche pas bien sur mon téléphone" | Non — c'est un bug, je corrige |

### Format de retour attendu

- Les retours sont envoyés **en une seule fois** par tour (pas au fil de l'eau).
- Format : document écrit (email, Notion, Google Docs) avec captures d'écran si nécessaire.
- Délai de retour : **[X] jours ouvrés** max par tour. Au-delà, la validation est considérée comme acquise.

### Exemple

> **Design** : 3 tours de révisions inclus.
>
> **Développement** : ajustements visuels pendant la recette (couleurs, espacements, tailles de texte).
>
> **Retours** : envoyés en une fois par tour, par écrit, sous 3 jours ouvrés. Passé ce délai, la version est considérée comme validée.
>
> **Hors périmètre** : toute demande d'ajout de section, de fonctionnalité ou de page non prévue dans ce document fera l'objet d'un devis complémentaire.

### Contenu

- **Canal principal** : par où on communique au quotidien (email, WhatsApp, Slack, etc.)

>

## Section 10 — Support post-livraison

**But** : Cadrer précisément ce qui se passe après la mise en ligne. C'est le sujet numéro un de malentendus post-projet.

### Contenu

- Durée du support inclus
- Volume horaire inclus
- Périmètre exact (ce qui est couvert et ce qui ne l'est pas)
- Délai de réponse pendant la période de support
- Ce qui se passe après la période de support

### Règles

- **Être chirurgical.** "1 mois de support" ne veut rien dire sans volume horaire et périmètre.
- Distinguer "correction de bug" (mon problème) et "évolution" (travail supplémentaire).
- Toujours proposer une option de maintenance continue — c'est du revenu récurrent et ça rassure le client.

### Exemple

> **Durée** : 30 jours calendaires à compter de la mise en ligne.
>
> **Volume inclus** : jusqu'à 3 heures de travail cumulées.
>
> **Couvert** :
> - Correction de bugs (dysfonctionnement par rapport à ce cahier des charges)
> - Modifications mineures de textes (fautes, reformulations)
> - Ajustements visuels mineurs (couleur, espacement, taille de police)
>
> **Non couvert** :
> - Ajout de nouvelles sections, fonctionnalités ou pages
> - Refonte ou modification substantielle du design
> - Problèmes liés à des services tiers (hébergeur, registrar, email)
>
> **Délai de réponse** : 48h ouvrées pendant la période de support.
>
> **Après les 30 jours** : toute intervention est facturée à [X]€/heure (devis préalable). Si vous souhaitez un accompagnement continu, je propose un forfait maintenance mensuel — on en parle à la livraison.

## Section 11 — Risques & hypothèses

**But** : Identifier ce qui pourrait bloquer ou ralentir le projet, et comment on y répond. Montrer du professionnalisme et se protéger en cas d'imprévu.

### Contenu

Un tableau des risques identifiés avec leur probabilité, leur impact et la mitigation prévue.

### Risques types

| Risque | Probabilité | Impact | Mitigation |
|--------|-------------|--------|------------|
| Retard de fourniture du contenu (textes, visuels) | Moyenne | Élevé — décale tout le planning | Deadlines claires en Section 5. Relance à J+2. Au-delà de [X] jours, reprogrammation. |
| Le client change d'avis sur la direction artistique après validation | Faible | Moyen — retravail DA | 3 tours de révision inclus. Au-delà, devis complémentaire. |
| Indisponibilité prolongée du client (vacances, urgence) | Faible | Élevé — bloque les validations | Identifier un interlocuteur de secours. Clause de reprogrammation. |
| Contenu insuffisant (pas de photos produit, pas de vidéo) | Variable | Moyen — dégrade la qualité | Alternatives identifiées (stock, illustrations). Coût additionnel si création nécessaire. |
| Problème technique tiers (API, hébergeur, service email) | Faible | Variable | Monitoring. Solution de fallback identifiée si possible. |

### Hypothèses de travail

> Ce cahier des charges repose sur les hypothèses suivantes. Si l'une d'elles s'avère incorrecte, le périmètre et/ou le planning seront réajustés d'un commun accord :
>
> - [Le client fournit des photos produit exploitables]
> - [Le nom de domaine est acheté et accessible avant la phase de mise en ligne]
> - [Un seul décisionnaire valide les livrables]
> - [Le contenu textuel est fourni en français — la traduction anglaise est à ma charge / co-rédigée]

### Règles

- Ne pas lister des risques absurdes ou anxiogènes. 3-5 risques réalistes et pertinents pour CE projet.
- Chaque risque a une mitigation. Pas de risque "on verra bien".
- Les hypothèses sont le filet de sécurité : si quelque chose change par rapport à ce qui a été convenu, ce document permet de dire "on avait posé cette hypothèse, elle a changé, donc on réajuste".

### Contenu

- Référence au devis signé (numéro, date)
- Rappel du montant et des modalités de paiement

- Si les conditions changent par rapport au devis initial (suite au cadrage), le mentionner ici avec l'accord des deux parties.
- **La clause de suspension protège les deux parties.** Elle n'est pas agressive — elle est professionnelle.

### Exemple

> Ce document complète le devis n°[REF] signé le [date].
>
> **Montant** : 2 500€ — Acompte de 1 250€ versé le [date], solde de 1 250€ à la livraison.
>
> **Changement de périmètre** : toute demande hors du périmètre défini dans ce document fera l'objet d'un devis complémentaire validé avant exécution.
>
> **Suspension** : si le projet est bloqué plus de 30 jours calendaires pour une raison indépendante de ma volonté (absence de réponse, non-fourniture de contenu, décision du client), je me réserve le droit de suspendre le projet. La reprise sera planifiée selon ma disponibilité. Les sommes déjà versées restent acquises au prorata du travail réalisé.

En cas d'annulation de mon fait, je rembourse les sommes correspondant au travail non effectué.

>
> **Portfolio** : je me réserve le droit de mentionner ce projet dans mon portfolio, sauf opposition écrite.

## Articulation des documents

Voici comment les 3 documents s'articulent dans le workflow commercial :

```
Prospect                     Client                      Projet terminé
   │                            │                              │
   │  ① Mise en relation        │  ② Devis/contrat             │
   │  (Template 1)              │  (justification prix)        │
   │  → Impressionner           │  → Engager                   │
   │  → Filtrer                 │  → Cadre légal               │
   │  → Préparer le RDV        │  → Prix, paiement, CGV       │
   │                            │                              │
   │                            │  ③ Cahier des charges        │
   │                            │  (Template 2 — ce document)  │
   │                            │  → Spécifier                 │
   │                            │  → Planifier                 │
   │                            │  → Référence pendant         │
   │                            │    toute l'exécution         │
   │                            │                              │
```

**Template 1** vend. **Le devis** engage. **Template 2** spécifie et guide l'exécution.

> **Note** : le devis (document ②) mériterait son propre template structuré. C'est le document légalement engageant — il devrait inclure au minimum : description des prestations, montant, modalités de paiement, délais, CGV, mentions légales obligatoires (SIRET, etc.), conditions de résiliation. La structure du devis doit être cohérente avec le Template 1 (reprendre le périmètre et le prix) et anticipée par le Template 2 (les conditions de référence y font écho).
